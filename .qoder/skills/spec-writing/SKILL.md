# 需求规格书撰写 (Spec Writing)

## 任务目标

将用户的原始需求（可能是口语化、不完整、跳跃的）转化为一份**可直接交给实现 Agent 的标准软件工程规格书**。

**禁止行为**：在规格书被用户确认前，不要直接写实现代码、不要直接改文件、不要直接执行 git 操作。

## 输出语言

所有输出使用**中文**。专有名词（API endpoint、函数名、类名、文件名）保留英文。

## 标准规格书结构

每份规格书必须包含以下 11 个章节。如果某个章节不适用（如纯前端 UI 任务无 API），也要显式写“N/A”并说明原因。

```markdown
# [任务编号/标题] 需求规格书

## 1. 任务摘要
- 一句话目标
- 所属模块/Agent
- 预计复杂度（高/中/低）

## 2. 背景与目标
- 为什么要做这个功能
- 解决什么痛点
- 成功标准是什么

## 3. 用户故事
按角色描述：
- 作为 [角色]，我希望 [功能]，以便 [价值]
- 至少覆盖主路径和 1 个次要角色

## 4. 验收标准 (Acceptance Criteria)
使用 Gherkin 风格或清单：
- Given [前置条件] When [动作] Then [预期结果]
- 至少包含：正常路径、错误路径、边界条件

## 5. 功能需求
拆分到不可再分的原子需求，每条格式：
- FR-XXX: [需求描述] (优先级: P0/P1/P2)

## 6. 非功能需求
- 性能要求
- 安全要求
- 兼容性要求
- 可观测性要求（日志/指标/追踪）

## 7. 数据模型设计
- 新增/修改的 Pydantic / SQLAlchemy / Vue 类型
- 字段名、类型、约束、默认值
- 关系图（用文字或 Mermaid 简单描述）

## 8. API / 接口契约
- 接口名称
- 输入参数（类型、必填、示例）
- 输出结构（类型、字段说明）
- 错误码与异常

## 9. 错误处理与边界条件
- 空输入/无效输入如何处理
- 依赖服务不可用如何降级
- 并发/竞态条件
- 权限/认证相关边界

## 10. 依赖与影响范围
- 依赖的已有模块/服务/工具
- 会影响的已有文件/表/API
- 是否需要迁移脚本或数据回填
- 需要新增的依赖库

## 11. 风险与缓解
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| ... | 高/中/低 | 高/中/低 | ... |

## 12. 参考实现线索
- 现有代码中类似功能的位置
- 可以复用的模块/工具/模式
- 建议的实现步骤（不需要写代码）
```

## 执行流程

### 步骤 1：澄清需求

如果用户的描述存在以下任何一种情况，**必须先提问澄清**，不要直接出规格书：

- 目标模糊（“优化一下”但没有具体指标）
- 范围不清（“加一个通知功能”但没有渠道/触发条件）
- 缺少角色/场景
- 涉及外部系统但没有说明接口
- 技术选型没有约束（如数据库、消息队列）

使用 1-3 个简短问题，每个问题给出 2-4 个选项让用户选择。

### 步骤 2：上下文分析

在写规格书前，必须分析项目上下文：

1. 读取 `AGENTS.md` 了解项目结构、命名规范、代码风格
2. 读取目标模块的 `__init__.py` 和已有 `models.py`
3. 如果涉及跨模块，读取相关模块的核心文件
4. 查看 `pyproject.toml` 确认依赖约束
5. 查看 CI 配置确认质量门禁

**原则**：不要一次性加载整个代码库。按需读取，最多 5-8 个文件。

### 步骤 3：撰写规格书

按上面的 12 章结构输出。每一章都要：

- **具体**：避免“处理好错误”这类空话，写成“当 x 为空时返回 400 并记录 warning 日志”
- **可验证**：每条验收标准都能通过测试或代码审查验证
- **无歧义**：一个需求只有一种理解方式
- **工程化**：考虑依赖、边界、回滚、监控

### 步骤 4：确认与迭代

规格书写完后，向用户确认：

1. “以上规格书是否准确反映了你的需求？”
2. “是否有遗漏的场景或边界条件？”
3. “是否可以直接进入实现阶段？”

用户确认后，输出一句过渡语：

> 规格书已确认。你可以继续让实现 Agent 按此规格书开发，或让我直接制定实施计划。

## 质量检查清单

在输出最终规格书前，自检：

- [ ] 用户故事覆盖了主要使用者和至少一个边缘角色
- [ ] 验收标准包含正常路径、错误路径、边界条件
- [ ] 数据模型字段有类型、约束、默认值
- [ ] API 输入输出有明确类型和示例
- [ ] 错误处理覆盖了空输入、无效输入、服务不可用
- [ ] 依赖分析列出了影响的现有文件/模块
- [ ] 风险评估至少包含 2 条
- [ ] 没有直接给出实现代码

## 示例：好 vs 坏的规格书片段

### 坏示例

```markdown
## 功能需求
- 加一个消息通知功能
- 支持多种渠道
- 要稳定可靠
```

### 好示例

```markdown
## 功能需求
- FR-001: 当分析任务完成时，系统自动向创建者发送通知 (P0)
- FR-002: 通知渠道支持站内信和邮件，后续可扩展 Webhook (P1)
- FR-003: 通知内容包含任务 ID、状态、完成时间、结果链接 (P0)
- FR-004: 通知失败时记录到 governance_agent 审计日志，不阻塞主流程 (P1)
```

## 注意事项

1. **不要让用户做你本该做的工作**。规格书的撰写是你的职责，用户只负责确认和补充。
2. **不要过度工程化**。如果用户明确说“先做个 MVP”，则 P2 需求可以延迟。
3. **优先用项目已有模式**。规格书中的设计要和 `AGENTS.md` 以及现有代码风格一致。
4. **如果需求明显超出项目当前阶段**，在风险章节指出，并建议拆分成子任务。
