# FDE 主计划 v3 · 软件工程与落地能力评审

> 评审对象：`master-delivery-plan.md`（2026-07-11 重构版）
> 评审视角：软件工程合规性 + 实际落地能力
> 评审日期：2026-07-11
> 评审依据：重读主计划全文 + 核实服务器资源（ARM 2C/11G）、测试规模（230 文件/800+ 用例）、CI 现状（ci.yml+deploy.yml）、迁移现状（无 Alembic，用 `schema.sql` 裸建表）

---

## 总体结论

**评分：B（可用但有关键落地风险）**

主计划在**战略层**做得好——契约先行、MVS 里程碑、降级方案、依赖标注都正确，比 v2 显著进步。但在**工程纪律层**和**物理约束层**有 3 个致命缺口，不修则「20 周交付」不可信。

| 维度 | 评分 | 说明 |
|------|------|------|
| 战略/优先级 | A- | 契约先行、MVS、降级方案 |
| 依赖管理 | B+ | 跨阶段依赖标注清晰，但跨仓库契约版本化缺失 |
| 测试策略 | **D** | 800+ 测试传统在新计划里**完全断裂**，零测试规划 |
| CI/CD 集成 | **C-** | 已有 CI 但计划未引用；新增 30+ 端点无门禁 |
| 迁移策略 | **D** | 计划写 Alembic 但项目无 Alembic；裸 SQL 无版本管理 |
| 安全深度 | **C** | ABAC 提了但文件上传/凭证存储/iframe 安全全缺 |
| 资源可行性 | **C-** | 单机 11G 已跑 12+ 容器，加 Redis+MinIO 未评估 |
| 人天现实性 | **C+** | solo 20 周×6 技术栈偏乐观；P6 规模化不现实 |

## P0 致命问题（不修则无法落地）

### P0-1：测试策略完全缺失，800+ 测试传统断裂

**问题**：当前项目有 230 个测试文件、800+ 用例、87% 覆盖率（M1-M4 积累）。但主计划 17 个阶段、新增 30+ HTTP 端点 + 10 个前端页面 + 4 张表 + 连接器适配器 + RAG 重排/量化，**全文没有一处提到单元测试/集成测试/E2E 测试的覆盖目标**。P0 的「评测集」只针对 RAG 检索质量（recall@k/MRR），不是软件测试。

**违反的工程原则**：测试金字塔——新功能必须有对应测试，否则无法安全重构和回归。

**影响**：
- H3 一次性补 5 个模块的后端端点（F/G/J/D/H），无测试 → 无法保证薄包装正确性
- P2 字段归一化引擎是核心逻辑，无测试 → 字段映射规则变更无回归保护
- P3 重排/改写改检索链路，无测试 → 量化（P4）时无法确认是否回归

**修复**：
- P0 新增交付物：**测试规约**（每个新端点 ≥1 集成测试；归一化引擎 ≥10 单元测试；RAG 链路 ≥5 E2E 测试）
- 每阶段验收标准加一条：**该阶段新增代码测试覆盖率 ≥80%**，CI 不通过不算交付
- H3（5 模块端点）拆分验收：每模块端点+测试独立交付，不积压

### P0-2：单机资源容量未评估，11G 内存撑不住目标架构

**问题**：服务器 ARM 2C/**11G**/96G。当前已运行：Dify 12 容器（约 3-4G）+ Qdrant（~1G）+ Postgres（~1G）+ fde-backend（~0.5G）+ nginx。主计划要新增：Redis + MinIO + ONNX 推理 + Docling（torch+transformers）+ BGE-Reranker（568M 参数）+ 门户前端构建。**全文无一处做内存容量评估**。

**违反的工程原则**：容量规划先于架构设计——在资源受限环境上叠加服务必须先算账。

**影响**：
- P0.5 spike 说「Docling ARM 可行性」，但即使解析可行，**常驻内存**是否可行没算
- P3 重排 BGE-Reranker-v2-M3（568M）在已满载的机器上可能直接 OOM
- P3b 同时加 Redis+MinIO，11G 机器可能撑不住
- P6「分布式 Qdrant」「Postgres 读写分离」在**单机**上根本无法实现

**修复**：
- P0 新增交付物：**资源容量评估表**（每新增服务的 CPU/RAM/磁盘预算 vs 当前可用）
- 明确硬约束：**ARM 2C/11G 单机上，Docling + Reranker 只能二选一常驻，或都走独立 GPU 微服务**
- P6 拆分：单机能做的（异步 worker、幂等、outbox）留下；需要多机的（分布式 Qdrant、读写分离）标记为「**需扩容前提**」，不排入当前 20 周

### P0-3：MVS 名不副实——38 人天不是「Minimum」

**问题**：§4.1 定义 MVS 横跨 H0→P0→H1→P1→H2→P2→H2b，累计 1+6+3+8+5+9+6 = **38 人天（约 8 周）**。这不是「最小可行切片」，这是半个项目。MVS 的价值在于**最快验证核心假设**，8 周才看到第一个可演示切片，反馈周期太长。

**违反的工程原则**：MVS 应是「最小成本验证最大不确定性」——这里最大不确定性是「字段归一化 + 入库 + RAG 命中」这条链，不需要门户已有能力页（H2）和连接器（P1）。

**修复**：定义**真正的 MVS（≤10 人天）**：
- 门户只做「登录 + 上传页 + 对话页」3 个页面（不做 10 菜单骨架）
- 后端只做 `/ingest/upload` + 字段归一化 + 入库 + RAG 查询
- **不含**连接器、不含已有能力页、不含权限打磨
- 验收：上传 1 份乱列名 Excel → 对话问答命中 → 证明「多源数据→统一入库→RAG」核心闭环
- 把当前的「38d MVS」降级为「**Milestone 1（M1）**」，MVS 是 M1 内的 10d 切片

## P1 严重问题（软件工程缺陷）

### P1-1：数据库迁移策略不完整——计划写 Alembic 但项目无 Alembic

**问题**：计划 P0 交付物写「4 表 Alembic migration」，但项目当前**无 Alembic 目录**，用 `shared/models/schema.sql` 裸 `CREATE TABLE IF NOT EXISTS` 建表。计划没提「先引入 Alembic 框架」这一前置工作，也没提现有表怎么纳入版本管理。

**影响**：P0 开始才发现没有迁移框架，要么临时引入（额外 1-2d 未排），要么继续裸 SQL（无版本管理、无回滚、生产改表危险）。

**修复**：
- P0 拆出子任务：**P0a 引入 Alembic + 将现有 schema.sql 纳入 baseline migration**（1d）
- P0 交付物改为「4 表 Alembic migration + **downgrade 脚本** + 生产迁移 runbook」
- 明确：生产 Postgres 已有数据（users/audit_logs 等），迁移必须 `stamp` 现有基线，不能 drop

### P1-2：文件上传安全方案完全缺失

**问题**：H2b 新增 `/ingest/upload` 端点接受 docx/pptx/pdf/xlsx/txt 上传，但计划**零字**提及：文件类型白名单校验、大小限制、恶意文件检测、路径穿越防护、临时文件清理。这是面向用户的上传入口，安全是刚需。

**修复**：
- H2b 新增安全子任务：① 文件类型 magic-bytes 校验（不只看扩展名）；② 大小限制（如 50MB）；③ 存储路径用 UUID 不用原文件名；④ 上传后病毒扫描钩子（预留）；⑤ 临时文件 TTL 清理
- 风险登记表新增「文件上传安全」

### P1-3：跨仓库契约版本化缺失

**问题**：ConnectorManifest 连接 FDE（Python）和 logistics（Java）两个独立仓库。计划定义了 manifest v1，但没提：manifest 有版本字段吗？FDE 收到 v2 manifest 怎么处理？logistics 改了 manifest 格式怎么保证 FDE 不崩？

**违反的工程原则**：跨系统契约必须版本化 + 向后兼容策略。

**修复**：
- `connector-contract.md` 新增 §版本化：manifest 含 `schema_version` 字段；FDE 按 semver 兼容；破坏性变更需双端同步发布
- P1 验收加一条：mock 连接器返回 v1 和「未知版本」manifest，FDE 能优雅降级

### P1-4：CI/CD 未纳入计划门禁

**问题**：项目已有 `.github/workflows/ci.yml`（lint→test）和 `deploy.yml`。但计划 17 个阶段无一处提到「CI 必须通过才能进入下一阶段」。新增 30+ 端点 + 前端项目，CI 配置需要更新（加前端 lint/build、加新端点的测试），计划没提。

**修复**：
- 每阶段验收加：「**CI 全绿**」作为 stage gate
- H1（门户骨架）新增子任务：更新 ci.yml 加入前端 lint+build
- P7 部署明确走 `deploy.yml` 自动化，不手动 scp

### P1-5：P6 规模化内容在单机上不现实

**问题**：P6（8d）塞了：异步 worker + 分布式 Qdrant + Postgres 读写分离 + GPU 推理微服务 + outbox 一致性 + 可观测。其中「分布式 Qdrant」和「Postgres 读写分离」**需要多节点**，ARM 2C 单机做不到。8 天做完这 6 项也不现实。

**修复**：
- P6 拆为 **P6a（单机可做）**：异步 worker + 批量嵌入 + outbox 一致性 + 可观测接入（4d）
- **P6b（需扩容前提，移出当前周期）**：分布式 Qdrant + 读写分离 + GPU 微服务——标记为「Phase 2（需扩容）」，当前 20 周不含
- 总人天从 102 → ~98（P6 8→4d，P6b 不计）

## P2 改进建议（提升落地质量）

### P2-1：文档应分阶段增量，而非 P7 集中补写

**问题**：P7（5d）才写文档。20 周项目中间不维护文档，到末尾集中补写是反模式——会遗忘细节、阻塞验收。

**修复**：每阶段交付物加「该阶段文档增量」（API 文档、运维 runbook 章节），P7 只做整合+复盘。

### P2-2：solo/团队两种口径应分开，`‖` 标注在 solo 下是噪音

**问题**：计划大量标注 `‖`（可并行）但反复说 solo 按顺序排。solo 下这些标注无意义且误导（暗示 16-17 周可达，但 solo 做不到）。

**修复**：排期表分两列——「solo 周次」和「3 人团队周次」，`‖` 只在团队列有意义。

### P2-3：H2b 两个菜单依赖不同，不应打包

**问题**：H2b 把「文档入库」（依赖 P2）和「连接器管理」（依赖 P1）打包。P1 早于 P2，连接器管理页被 P2 拖延。

**修复**：连接器管理页移到 H2（只依赖 P1），文档入库页留 H2b（依赖 P2）。

### P2-4：缺少阶段门禁（Stage Gate）机制

**问题**：计划有降级方案，但没说「阶段交付物不合格时怎么办」——是回滚、重做、还是继续带病前进？

**修复**：每阶段结束设 stage gate：① CI 全绿；② 验收标准达成；③ 评测不降级。不通过则暂停调整，不进入下一阶段。

### P2-5：可观测性接入计划不完整

**问题**：H2 做了监控页（Grafana iframe），但新增 30+ 端点 + 10 页面的日志/metrics/trace 怎么接入已有 OTel/DecisionChainLog 没提。前端错误上报也没提。

**修复**：H3 新增子任务——所有新端点自动注入 trace_id（复用 TracingMiddleware）；前端加错误上报（Sentry/自建）。

### P2-6：凭证存储加密方案缺失

**问题**：连接器管理页要配置各平台 AppID/Secret/Token，门户 IM 页也要配凭证。计划没提这些敏感凭证怎么存——明文进 Postgres？加密？用环境变量？

**修复**：P0 契约定型新增「凭证存储规范」——Postgres 加密列（pgcrypto）或外部 secret manager，不明文存储。

## 一句话总结

**战略层 A-，工程纪律层 C。** 最该立刻修的三件事：① 补测试规约（别让 800+ 测试传统断裂）；② 算资源账（11G 单机撑不住目标架构，P6 要拆）；③ 把 MVS 砍到 10 天以内（38 天不是 minimum）。修完这三条，计划从「PPT 可信」变成「可执行」。
